## 2. Priority check: no hidden latest-contact response path

The possible confound was:

```text
H_P -> materialization packet/contact identity -> native feedback
```

rather than the declared:

```text
H_P -> R_H -> M_H live state -> native feedback
```

The frozen harness calls one neutral native contact after every final
materialization and before feedback emission. Across the `q1-q2` and `q2-q1`
order branches, the latest source contact matches on:

- physical source and target;
- contact amount `0.01`;
- packet-arrival event kind;
- route and channel;
- event time `8.0`;
- scheduler index `9`; and
- neutral lineage/configuration.

The retained latest-contact rows differ only in node-proper-time assignment
and the two digests derived from that assignment. That difference honestly
records the earlier physical event order. It does not enter the response
decision:

1. `emit_feedback_eligibility_surface_row()` calculates
   `polarity_score = (front_mass - rear_mass) - reference` from the live
   `M_H/B_ref` mask.
2. It carries latest-contact identity into the feedback row for evidence and
   lineage, but does not use contact amount, endpoints, node proper time,
   route, channel, or source digest in the polarity calculation.
3. The producer reads `boundary_polarity_score`, expected polarity, threshold,
   response route/reserve, and current-lineage/idempotency guards.
4. `expected_source_surface_digest` is frozen `null`, so differing source-row
   digests cannot authorize success.
5. Both order branches use a current fixed-topology feedback row and a fresh
   producer idempotency state.

The retained order observations therefore remain attributable to different
`M_H` values (`0.5` versus `0.4`) produced by different H_P readouts, not to a
latest materialization contact.

## 3. Priority check: functional active-history identity

The functional pool is frozen as:

```text
H_P persistent active history + declared M_H native output-port binding
```

It is neither native intake state P nor M_H alone.

`H_P` is adapter-owned persisted state containing the ordered token tuple,
cursor/idempotency state, readout configuration, materialization state,
intervention provenance, and reset baseline. `R_H` reads the tuple itself.
`M_H` is a reconstructible native materialization/cache of that readout and is
insufficient restoration identity on its own.

The clamp branch changes active `H_P` through `replace_history()`, retains its
cursor and native contribution audit, recomputes `R_H`, and only then uses
public native balancing packets to rematerialize M_H. No later ingestion occurs
before feedback, so the clamp remains active through the response window. It
does not write a precomputed scalar or alter feedback/response state directly.
