### 2.1 LGRC9V3 substrate boundary

`GRC9V3.step()` advances one full synchronous hybrid step, including its own
continuity and other per-step state evolution. `LGRC9V3.step()` processes one
deterministic queue event and explicitly does not call the synchronous GRC
step. The resulting evolution and relaxation semantics are not interchangeable.

Consequently:

```text
eligible core substrate:
  LGRC9V3

GRC9V3 role:
  interesting comparative result only

forbidden inference:
  GRC9V3 success
  -> LGRC9V3 implementation
  -> L03 core support
```

`GRC9V3State` remains in the admitted runtime set because it is the public
base-state type used by LGRC9V3. That shared state type does not make the two
step semantics equivalent.

## 6. RCAE producer boundary

The admitted predecessor patterns show that an RCAE-owned completion can be
implemented without changing PyGRC:

```text
RCAE request producer
  observes only declared LGRC9V3 runtime-visible state
  emits an auditable request for public LGRC9V3 work

LGRC9V3
  owns every invoked native transition

optional RCAE declared-state producer
  owns only explicitly registered external state and update law
```

This is permission to consider a later implementation class, not selection of
one. The producer may not write the desired route, choose the successful
outcome, conceal a global selector, or weaken the accepted discriminator.
It also may not call synchronous GRC9V3 evolution and relabel the result as an
LGRC9V3 completion.

All bridging code must live in RCAE. It may import and call the admitted
PyGRC API, but it may not edit, monkey-patch, or silently extend the graph
repository. If PyGRC later implements a load-bearing RCAE pattern natively,
the retained producer continues to identify the historical implementation
until an explicit source/realization transition and rerun replace it.
